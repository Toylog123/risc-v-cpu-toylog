# YH_rv_cpu 技术文档

> 更新时间：`2026-03-17`
> 适用范围：当前仓库中的正式比赛工程 `YH_rv_cpu`
> 用途：作为整体设计总览、后续改动入口说明、协作和交接时的统一技术基线

## 1. 文档定位

这份文档不是简单的项目简介，而是当前 `YH_rv_cpu` 的技术总说明。

它解决的是 4 个实际问题：

- 任何时候都能快速看清当前 CPU/SoC/FPGA 到底做到哪一步
- 想改某一块设计时，能直接定位到应该改哪些文件
- 想判断某个改动会影响哪里时，有一个统一的架构视图
- 换设备、换智能体、换队友接手时，不需要从头猜设计结构

如果后续文档越来越多，这份文档应始终保持为“先看这一份就能恢复整体设计”的入口。

## 2. 当前项目目标

`YH_rv_cpu` 面向七星微赛题，目标不是做一个只在仿真里能跑的教学 CPU，而是逐步形成一个：

- 有完整 RTL
- 有验证链
- 有 FPGA 闭环
- 有文档和交接体系
- 能用于比赛提交和展示

当前路线已经明确：

- 不再做赛题切换
- 不再扩散到别的无关方向
- 主线是 `CPU -> 最小 SoC -> 验证 -> 跑分 -> FPGA -> 比赛材料`

## 3. 当前技术状态总览

### 3.1 当前已经完成的主线能力

- 五级流水 CPU 基线已经建立
- 数据通路已抽出 `XLEN` 参数化骨架
- 当前主验证基线是 `RV32I + Zicsr`
- 已支持最小机器态 `CSR / trap / machine timer interrupt`
- 最小 SoC 已打通
- UART 启动输出已能在仿真中跑通
- `XLEN=64` 基础烟测已建立
- `riscv-tests` 子集已接入并通过最小回归
- Vivado 本地综合链已打通

### 3.2 当前仍未完成的关键项

- `RV64` 指令语义还没有补齐到稳定可回归的程度
- `riscv-tests` 还只是子集，不是全量
- `CoreMark` 已接入框架，但还没形成稳定跑分链
- 当前 FPGA 约束仍是模板，不是实物板卡冻结版
- 当前 100MHz 时序未收敛
- 还没有真实板卡上的串口演示、bitstream 记录和板级日志

### 3.3 当前阶段的真实重点

当前不是“继续堆更多新模块”的阶段，而是“把已有主线做扎实”的阶段。

具体就是：

1. 补 `RV64`
2. 扩大验证
3. 跑通 `CoreMark`
4. 推进 FPGA 时序和板级闭环

## 5. CPU 顶层微架构

CPU 顶层文件是 [YH_rv_cpu.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu.v)。

它不是“纯组合拼装”，而是包含了完整的流水寄存器、CSR 状态、trap 重定向和写回流程。

### 5.1 流水级划分

- `IF`
  - 负责当前 `PC` 取指
  - 根据重定向结果决定 `pc_next`
- `ID`
  - 对 32 位指令做译码
  - 读寄存器堆
  - 生成立即数和控制信号
- `EX`
  - 进行 ALU 运算
  - 完成分支判断
  - 形成访存地址
  - 形成 store data / strobe
  - 检测 misaligned 和 illegal
- `MEM`
  - 连接 SoC 数据总线
  - 把 `load` 数据按 byte/half/word/dword 重新整理
- `WB`
  - 在 `ALU / MEM / PC+4` 之间选择最终写回数据

### 5.2 顶层内的关键流水寄存器

在 [YH_rv_cpu.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu.v) 中，流水寄存器是直接在顶层维护的：

- `if_id_*`
- `id_ex_*`
- `ex_mem_*`
- `mem_wb_*`

这意味着：

- 后续如果要增加性能统计、debug trace、额外控制字段，最直接的入口就是顶层这些寄存器
- 如果要做更深层的流水控制，例如更复杂的 flush/bubble 规则，也仍然是从顶层开始改

### 5.3 当前控制优先级

当前顶层的控制优先级核心是：

1. `rst_n`
2. `trap_r`
3. `ex_trap_valid`
4. `ex_mret_valid`
5. `ex_redirect_valid`
6. `stall_decode / stall_fetch`

也就是说，当前实现里：

- trap / interrupt / mret 的优先级高于普通 branch/jump redirect
- redirect 会冲刷 `IF/ID` 和 `ID/EX`
- `load-use hazard` 会暂停取指和译码，并在执行级插入 bubble

这条规则后续如果要变，必须优先审查：

- [YH_rv_cpu.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu.v)
- [YH_rv_cpu_hazard_unit.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_hazard_unit.v)

## 6. IF / ID / EX / MEM / WB 详细说明

### 6.1 IF 级

文件：

- [YH_rv_cpu_if_stage.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_if_stage.v)

当前行为：

- `imem_addr = pc_current`
- `pc_plus_4 = pc_current + 4`
- `pc_next = redirect_en ? redirect_pc : pc_plus_4`

这一级很薄，说明当前取指逻辑还没有引入：

- 指令 cache
- branch predictor
- fetch queue
- 对齐缓冲

所以后续如果要做性能优化，`IF` 级是一个天然扩展点。

### 6.2 ID 级

文件：

- [YH_rv_cpu_id_stage.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_id_stage.v)
- [YH_rv_cpu_decoder.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_decoder.v)

`ID` 级做两件事：

- 计算 `pc4`
- 通过 `decoder` 生成完整控制信号

`ID` 级本身非常轻，主要复杂度都压在 `decoder`。

### 6.3 EX 级

文件：

- [YH_rv_cpu_ex_stage.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_ex_stage.v)
- [YH_rv_cpu_alu.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_alu.v)

`EX` 级当前负责：

- ALU 运算
- branch compare
- jal/jalr redirect
- load/store 地址生成
- misaligned 检测
- word-op 下的 32 位结果再符号扩展

当前 `word_op` 的实现方式是：

- 先用 `word_result[31:0]`
- 再扩展成 `word_result_sext`

这对后续支持 `RV64I` 的 `*_W` 类指令是关键基础。

### 6.4 MEM 级

文件：

- [YH_rv_cpu_mem_stage.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_mem_stage.v)

`MEM` 级当前负责：

- 直接把 `mem_addr` 送给 SoC
- 直接把 `store_data` / `store_wstrb` 送出去
- 根据 `mem_size` 和 `mem_unsigned` 重组 load 数据

当前设计重点：

- `byte_offset` 是从地址低位推出来的
- `shifted_rdata = dmem_rdata >> {byte_offset, 3'b000}`
- 然后再按 `B/H/W/D` 做截取或符号扩展

这使得当前 load 路径已经具备：

- `LB/LBU`
- `LH/LHU`
- `LW`
- `LD`
- `LWU`

其中 `LD/LWU` 只有在 `XLEN==64` 且译码放行时才有效。

### 6.5 WB 级

文件：

- [YH_rv_cpu_wb_stage.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_wb_stage.v)

当前写回源只有 3 类：

- `ALU`
- `MEM`
- `PC+4`

这意味着当前设计里 CSR 指令的写回并没有单独走一条 `WB` 类型，而是把 CSR 读值在 EX 级就并入 `exec_result_final`。

这个细节后续如果要改 CSR 流水边界，必须注意。

## 7. 寄存器堆、前递和暂停

### 7.1 寄存器堆

文件：

- [YH_rv_cpu_regfile.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_regfile.v)

当前特征：

- `x0` 恒为 0
- 同周期写回旁路已实现
- 读端口是组合读
- 写端口是时钟上升沿写

当前旁路逻辑非常重要：

- 如果 `rd_wen && rd_addr == rs1_addr/rs2_addr`
- 则译码级直接看到 `rd_wdata`

这修掉过“同周期第一次读取字符串拿不到写回值”的问题，所以后续不要轻易删除。

### 7.2 hazard 处理

文件：

- [YH_rv_cpu_hazard_unit.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_hazard_unit.v)

当前只做了两类事情：

- `load-use` hazard 检测
- `EX/MEM`、`MEM/WB` 两级前递选择

当前没有做：

- 多周期运算暂停
- 更复杂的结构冒险处理
- store data 特殊前递控制
- 分支预测相关 hazard

所以后续如果引入：

- `M` 扩展
- cache
- 更复杂中断入口

hazard unit 很可能要同步扩展。

## 9. CSR / Trap / Interrupt 设计

### 9.1 当前支持的 CSR

在 [YH_rv_cpu.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu.v) 中，当前维护的 CSR 有：

- `mstatus`
- `mie`
- `mtvec`
- `mscratch`
- `mepc`
- `mcause`
- `mip` 以组合方式由 `timer_irq` 反映

### 9.2 当前支持的 trap 指令和事件

- `ecall`
- `ebreak`
- `mret`
- illegal instruction
- load/store misaligned
- machine timer interrupt

### 9.3 当前 trap 重定向规则

- trap / interrupt 进入 `mtvec`
- `mret` 返回 `mepc`
- `mepc` 保存当前 `id_ex_pc_r`
- `mcause` 保存 trap cause
- `mstatus.MIE/MPIE` 做最小机器态切换

### 9.4 当前实现边界

当前 trap 体系是“够比赛主线”的最小机器态，不是完整 privileged 架构。

还没有做：

- user/supervisor 模式
- page table
- delegation
- 外部中断控制器
- 异常精细优先级系统

## 10. SoC 设计

文件：

- [YH_rv_cpu_soc.v](d:/BaiduSyncdisk/secretary/竞赛/集创赛/YH_rv_cpu/rtl/YH_rv_cpu_soc.v)

### 10.1 当前 SoC 组成

- `ROM`
- `RAM`
- `UART TX`
- `DONE`
- `timer`

### 10.2 当前地址映射

- `ROM_BASE = 0x0000_0000`
- `RAM_BASE = 0x0000_4000`
- `UART_TX_ADDR = 0x1000_0000`
- `DONE_ADDR = 0x1000_0004`
- `TIMER_VALUE_LO = 0x1000_0008`
- `TIMER_VALUE_HI = 0x1000_000C`
- `TIMER_CMP_LO = 0x1000_0010`
- `TIMER_CMP_HI = 0x1000_0014`
- `TIMER_CTRL_ADDR = 0x1000_0018`

### 10.3 当前 ROM/RAM 实现

当前实现仍然偏“先打通工程”的策略：

- `ROM`：字节数组
- `RAM`：32 位 word 数组

本地综合结果显示：

- `ROM` 目前仍主要映射成 LUT ROM
- `RAM` 目前主要映射成 distributed RAM

所以后续如果目标是：

- 降低 LUT 占用
- 改善时序

最优先的方向之一就是把 `ROM/RAM` 往 BRAM 方案推进。

### 10.4 UART / DONE / timer

当前 UART 是最小发送路径，只负责把 SoC 输出字节序列发出去。

`DONE` 用来标识程序完成。

`timer` 维护：

- `timer_value_r`
- `timer_cmp_r`
- `timer_irq_en_r`

并通过：

- `timer_irq = timer_irq_en_r && (timer_value_r >= timer_cmp_r)`

向 CPU 提供 machine timer interrupt 条件。

## 19. 2026-03-17 时序试探与 BRAM 迁移结论

这一轮额外做了一次针对 `100MHz` 的试探性 RTL 优化，重点是把 `IF/ID`、`ID/EX` 在 `flush/stall` 时的寄存器使能链拆开，观察能否降低 EX 到流水寄存器的控制关键路径。

### 19.1 这次试探做了什么

- 尝试把 `IF/ID`、`ID/EX` 的 `flush/stall` 处理改成更偏“显式 bubble / 重新采样”的写法
- 目标是减少综合后落在寄存器 `CE` 引脚上的组合控制链
- 实际上这组改动没有保留，已经回退到当前稳定基线

### 19.2 为什么没有保留

试探过程中得到过两组更差的 100MHz 结果：

- 一次最差到 `WNS = -3.139ns`
- 另一次回到 `D` 端路径后仍有 `WNS = -2.938ns`

而当前稳定基线重新综合后的结果是：

- `100MHz`：`WNS = -2.487ns`

所以当前结论很明确：

- 简单改 `CE` / `bubble` 写法，不足以把 100MHz 时序真正拉回来
- 这类小修小补可能改变最坏路径位置，但没有带来实质收敛
- 现阶段不值得把这条试探性 RTL 留在主线里

### 19.3 为什么当前还没有用上 BRAM

这不是单纯的 `ram_style` / `rom_style` 属性问题，而是当前 SoC 存储接口本身就是“零等待、组合读出”的结构。

当前真实行为是：

- `imem_rdata` 直接由 `rom_mem[...]` 组合拼出来
- `dmem_rdata` 直接由 `ram_mem32[...]` / MMIO 组合给出
- `load` 和取指都默认同周期拿到返回值

这决定了当前综合倾向于：

- `ROM` 落到 LUT ROM
- `RAM` 落到 distributed RAM / LUTRAM

所以现在 `BRAM = 0` 不是工具偶然没推出来，而是当前接口语义本来就更接近异步读存储。

### 19.4 如果要真正推进 BRAM 化，必须先改什么

后续要把 `ROM/RAM` 往更适合 FPGA 的 BRAM 方案推进，优先顺序应该是：

1. 抽象指令存储器响应时序，让 `IF` 能接受同步读返回
2. 给数据存储器引入明确的一拍返回语义，或在 `MEM` 级加等待/响应处理
3. 把当前 SoC 中直接组合读取数组的写法，改成独立的存储包装模块
4. 在存储包装模块里再切 `block` 风格或厂商原语/模板

也就是说：

- 先改接口语义
- 再改存储实现

如果顺序反过来，往往只会得到“工具继续用 LUT 实现”或者“功能被破坏”的结果。

### 19.5 当前最值得做的下一步

从今天这轮试探之后，FPGA 线的下一步更清楚了：

1. 保持当前稳定基线，不把无收益的时序试探留在主线
2. 先把 `ROM/RAM` 改造拆成一个单独工作包，明确接口改动范围
3. 再决定是优先做同步 `imem`，还是先做同步 `dmem`
4. 继续用 `50MHz` 作为当前比赛可交付频率口径，`100MHz` 作为优化目标
