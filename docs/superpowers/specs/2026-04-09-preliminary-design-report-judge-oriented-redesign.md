# Preliminary Design Report Judge-Oriented Redesign

## Goal

把 `01-项目管理/04-汇报与提交材料/初赛提交材料/latex/` 下的初赛设计说明书从“内部交接式工程文档”收敛成“面向赛事评审的正式设计说明书”，突出工程完整性、架构清晰度和验证可信度，同时修复当前 PDF 已出现的表格重叠与章节定位偏差。

## Current Problems

1. 现有文稿混入了内部项目管理口吻，例如“材料组织方式”“内部支撑文件目录”等内容，不适合直接面向裁判。
2. 当前表格使用固定列宽且包含较长文件路径，导致 PDF 中出现内容重叠。
3. 插图仍是占位框，影响正式感和技术表达力度。
4. 章节叙事缺少明确主次，工程可信度、架构设计和验证结果三类信息没有形成统一的裁判阅读路径。

## Target Positioning

说明书的主叙事为：

- 先建立信任：这是一个完整、可信、可复现的比赛工程；
- 再建立技术说服力：关键微架构设计清晰且取舍合理；
- 最后用结果收束：验证、性能、资源、FPGA pre-board 路径都有证据支撑。

## Required Changes

### 1. Chapter Structure

保留以下正式章节：

1. 封面、摘要、关键词
2. 赛题理解与设计目标
3. 总体方案与关键微架构设计
4. 验证方案与实验结果
5. FPGA 原型适配情况
6. 总结与后续工作
7. 附录：关键结果摘要

必须移除或改写以下内部化内容：

- “初赛提交材料组织方式”
- 对内部目录结构的介绍
- 面向交接者而非评审者的路径导航
- 过强的“文档管理流程”叙述

### 2. Table Policy

- 正文表格只保留裁判真正关心的技术信息，不直接堆叠长源码路径。
- 模块表改为“模块名/功能说明/设计要点”而不是“文件路径/作用”。
- 结果表使用可伸缩列宽，避免再次出现重叠。
- 长路径只放在附录中，且以摘要式证据索引呈现。

### 3. Figure Policy

需要补入 3 张正式技术图：

1. 系统总体架构图：CPU 核、SoC、FPGA 原型顶层关系。
2. 五级流水与关键控制通路图：数据通路、前递、load-use stall、redirect/trap。
3. 验证闭环图：Smoke、baseline、full-ui、CoreMark、impl50、FPGA-like probe 之间的关系。

图的风格应为工程技术图，而不是装饰性插图。

### 4. Tone Policy

- 保持正式、克制、工程化。
- 可以说明边界，但不能写成“内部工作记录”。
- 可以强调验证与工程闭环，但不渲染过程管理细节。
- 可以说明未完成项是外部阻塞，但要聚焦“对设计提交的影响”，而不是项目协作流程。

## Acceptance Criteria

1. PDF 中不再出现表格内容重叠、溢出或路径覆盖。
2. 正文不再出现“材料组织方式”“内部支撑材料”“交接说明”等评审无关内容。
3. 至少 3 处占位图被正式技术图替换。
4. 评审阅读顺序清晰：设计目标 -> 架构 -> 验证结果 -> FPGA 适配 -> 总结。
5. 正式结果与仓库当前真实冻结基线保持一致，不引入未经验证的新口径。

## Constraints

- 不修改技术结果本身，只重构说明书表达方式和版式。
- 不把当前未收口的 profiling/优化支线内容误写成正式结论。
- 任何相关 README 或说明文档同步时，只做与本次正式化收口直接相关的更新。
