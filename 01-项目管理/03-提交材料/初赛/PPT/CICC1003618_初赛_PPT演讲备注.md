## 第 1 页：封面

各位评委好，我们的作品是“基于 RISC-V 的五级流水 CPU 设计与 FPGA 验证”。本作品面向第十届全国大学生集成电路创新创业大赛初赛，队伍编号是 CICC1003618，团队名称是芯银河战队。

本次汇报会围绕两个重点展开：第一，说明这颗 CPU 的架构设计、流水线控制、指令扩展和低资源实现方式；第二，展示仿真、性能基准、资源时序、PYNQ-Z2 上板和 UART 实时输出形成的完整验证证据链。

## 第 2 页：快速预览 1 - 关键指标

这一页先给出作品的核心结果。我们的设计是一颗面向 Xilinx PYNQ-Z2 FPGA 原型系统的轻量级高性能 RISC-V 软核 CPU。

在冻结提交路径下，CPU 运行在 50.0 MHz。CoreMark 成绩为 4.137461 CoreMark/MHz，Dhrystone 成绩为 2.908287 DMIPS/MHz。实现后资源占用为 4961 个 LUT，满足 LUT 小于 5000 的硬性约束。

在上板验证方面，设计已经完成 PYNQ-Z2 的 bitstream 下载，Vivado 日志显示 PROGRAM_OK，并且通过 PL UART 连续输出 YH_rv_cpu 的 CoreMark、DMIPS、tick 和 PC 信息。也就是说，指标不是只停留在文档里，而是由仿真、性能日志、Vivado 报告和上板串口输出共同支撑。

## 第 3 页：快速预览 2 - 核心亮点

这一页概括作品的四个核心亮点。

第一，指令热点增强方面，在 RV32I 基础上加入了 Zmmul 乘法扩展，以及 Zba、Zbb、Zbs 位操作相关扩展。这些扩展对应 CoreMark 和 Dhrystone 中常见的乘法、地址生成和位操作热点。

第二，流水线控制方面，采用五级顺序流水，并对跳转、分支、前递、停顿和刷新进行了协同优化。JAL early redirect 能减少无效取指，forwarding 和 stall/flush 逻辑用于保证数据相关和控制相关场景下的正确性。

第三，低资源低功耗取向方面，我们没有采用乱序执行、复杂动态预测或完整除法器这类高成本结构，而是把资源集中在热点路径上，保证提交路径满足 5000 LUT 内的约束。

第四，工程验证方面，形成了从 RTL、TestBench、CoreMark、Dhrystone，到 Vivado 实现、PYNQ-Z2 上板和 UART 输出的闭环证据。

## 第 4 页：赛题理解与设计目标

这一页说明我们的设计目标来自赛题约束，而不是单纯追求某一个跑分。

初赛要求 FPGA CPU 原型能够稳定运行，支持 RV32I 或 RV64I 指令集之一，主频不低于 50 MHz，并提交仿真与 FPGA 测试数据。同时，源码需要结构完整、可复现，代码注释率不低于 30%，并且本赛题对资源，尤其是 LUT 规模有明确限制。

基于这些要求，我们选择 RV32I 五级顺序流水作为主体架构，适配 PYNQ-Z2 板卡，冻结提交时使用 50 MHz CPU 时钟，并用 CoreMark 和 Dhrystone 两个基准分别观察综合性能和整数性能。对于扩展功能，我们采用参数化方式进行筛选：能提升热点性能并且满足资源时序约束的进入提交路径，超出资源约束的高性能探索路径只作为后续方向。

所以我们的设计原则是，先保证可综合、可验证、可上板，再对关键热点路径做选择性增强。

## 第 5 页：总体架构

这一页是系统总体架构。

最外层是 PYNQ-Z2 FPGA 顶层，顶层模块负责板级时钟、复位、Pmod UART 和 LED 连接。往里是 SoC 层，SoC 层负责片上 ROM、RAM 和 MMIO 映射，让 CPU 可以访问程序、数据和外设。核心部分是 YH_rv_cpu，它采用 IF、ID、EX、MEM、WB 五级流水结构。

指令流从 ROM 进入取指阶段，数据访问通过 RAM 和 MMIO 完成，外设输出主要包括 UART TX 和 LED。UART 用于上板时实时打印运行状态，LED 用于快速观察板级活动状态。

通过这样的分层，CPU Core、SoC 和 FPGA 顶层的职责比较清晰：CPU 负责执行与控制，SoC 负责存储和外设通路，FPGA 顶层负责把设计落到 PYNQ-Z2 的真实引脚和时钟环境中。

## 第 6 页：五级流水与控制通路

这一页展示五级流水和核心控制通路。

CPU 流水线分为 IF、ID、EX、MEM、WB 五个阶段。IF 负责取指，ID 负责译码和寄存器读取，EX 负责 ALU、分支判断和乘法等执行逻辑，MEM 负责访存，WB 负责写回。

流水线正确性的关键在于三个控制动作：stall、flush 和 forwarding。stall 用于必要停顿，flush 用于清除错误路径上的指令，forwarding 用于把后级结果尽早送回前级，减少不必要的等待。

在跳转和分支方面，我们采用“提前判断加后级校验”的思路。ID 阶段进行轻量的跳转或分支提前处理，尽量减少前端空转；EX 阶段保留校验和兜底重定向，保证复杂场景仍然正确。这样可以在保持结构简单的同时，降低控制相关对性能的影响。

## 第 7 页：ISA 扩展与热点优化

这一页把提交路径中的 ISA 扩展和性能热点对应起来。

基础指令集是 RV32I。为了提升常见整数程序的热点性能，我们在提交路径中加入了 Zmmul、Zba、Zbb 和 Zbs。

Zmmul 主要面向乘法热点。没有硬件乘法时，乘法往往需要较长的软件序列展开；加入 Zmmul 后，可以用更直接的硬件路径处理乘法相关计算。

Zba、Zbb、Zbs 主要面向地址生成、位操作和位提取等热点。这些操作在 CoreMark 和 Dhrystone 这类基准中经常出现，硬件支持可以减少指令序列长度，降低流水线执行压力。

另外，我们也保留了若干参数化探索路径，比如更激进的位操作、分支和预测相关探索。但初赛冻结提交只采用资源和时序都满足约束的路径，避免用超资源方案夸大提交能力。

## 第 8 页：低功耗与低资源设计取向

这一页说明我们的 PPA 取舍，也就是性能、面积和功耗之间的平衡。

在 FPGA 资源受限的条件下，如果直接加入复杂动态预测、乱序执行、完整除法器或过宽旁路网络，虽然某些场景可能会提高性能，但 LUT、时序和调试复杂度都会明显上升，不利于稳定上板和复现。

因此，我们采用更克制的轻量化方案：主体保持顺序五级流水，把硬件增强集中在乘法、位操作、跳转重定向和必要的数据前递上；同时通过参数化开关控制不同扩展模块，便于在性能和资源之间筛选合适组合。

最终提交路径的目标不是堆资源，而是在 LUT 小于 5000、CPU 50 MHz、可上板和可复现的条件下，把热点性能尽可能做高。

## 第 9 页：RTL 建模规范与工程组织

这一页展示工程组织和建模规范。

源码目录分为五类。rtl 目录保存 CPU、SoC 和 FPGA 相关 RTL 设计代码；tb 目录保存模块级和系统级 TestBench；sw 目录保存 RISC-V 测试程序、CoreMark 和 Dhrystone 相关程序；fpga 目录保存 PYNQ-Z2 约束、Vivado 脚本和板级适配文件；scripts 目录保存构建、仿真、统计和日志整理脚本。

这种目录划分的目的，是让评审能够直接找到设计代码、测试代码、软件程序和 FPGA 复现入口。源码 README 和运行环境说明中也列出了工具版本、运行方式和目录职责。

从代码质量看，保守提交口径统计的源码注释率为 31.87%，RTL、TestBench 和 Verilog 口径注释率为 30.22%，满足赛题提出的不低于 30% 的要求。

## 第 10 页：验证体系

这一页说明我们的验证不是只跑一个 benchmark，而是分层闭环。

最底层是指令和模块定向测试，覆盖 Zmmul、Bitmanip、分支、Load-use、Trap、Timer 等关键功能点。第二层是 SoC smoke 测试，用于确认存储、MMIO 和基础系统通路能够跑通。第三层是性能基准，包括 CoreMark 和 Dhrystone，并对优化前后结果进行对比。第四层是 FPGA 实现与下载，包含 Vivado 资源时序报告、bitstream、PYNQ-Z2 下载日志和 UART 实时输出。

这四层验证分别回答四个问题：指令功能是否正确，系统通路是否可运行，性能数据是否可测量，FPGA 原型是否能真实落板。

## 第 11 页：关键波形与时序证据

这一页展示关键控制机制的波形证据。

第一类波形是 ID 分支快速重定向。观察点是 Fetch Redirect 拉高，而 EX Redirect 在对应场景下保持低，说明前端能够提前修正 PC，减少错误路径取指。

第二类波形是 JAL 快速重定向。这里重点看跳转目标是否能提前进入取指路径，以及流水线是否按预期刷新无效指令。

第三类波形是 Load-use 场景。这里重点观察 Stall Decode、前递和等待逻辑，确认在相关场景下控制信号没有产生错误停顿，也没有破坏数据正确性。

这些波形和时序描述对应验证报告中的截图和日志，用于说明优化逻辑不仅写在 RTL 中，而且在仿真中可以被观测和解释。

## 第 12 页：性能结果

这一页集中展示两个性能基准结果。

CoreMark 结果为 4.137461 CoreMark/MHz，对应测试日志中的 2416941 ticks 和 413.746136 iterations per second。Dhrystone 结果为 2.908287 DMIPS/MHz，对应 510986 Dhrystones per second，并完成 10 轮运行。

需要说明的是，CoreMark 和 Dhrystone 是分别独立运行的两个基准，不互相换算，也不互相外推。CoreMark 更偏向综合整数和控制流热点，Dhrystone 更偏向经典整数路径和库函数相关路径。两个指标共同用于观察 CPU 在不同程序形态下的表现。

这些数据的来源包括本地仿真日志、性能测试脚本输出和性能与验证报告，PYNQ-Z2 上板 UART 也会实时打印冻结口径下的 CoreMark/MHz 和 DMIPS/MHz。

## 第 13 页：优化路径与收益拆解

这一页说明性能是怎样一步步提升的。

早期 RV32I smoke 基线的 CoreMark/MHz 约为 0.912472，主要用于确认 CPU 和 SoC 能够跑通。随后经过流水线控制、热点扩展和构建路径优化，可提交参考路径达到 4.060276 CoreMark/MHz。最终冻结提交路径达到 4.137461 CoreMark/MHz，同时满足 4961 LUT、50 MHz 和上板复现要求。

DMIPS 的提升更明显。早期 Dhrystone 为 1.015253 DMIPS/MHz，冻结提交达到 2.908287 DMIPS/MHz，提升约 186.45%。这部分收益主要来自 Dhrystone 构建与库函数路径优化，以及 Zmmul、Bitmanip、JAL early redirect、forwarding 和 stall/flush 协同带来的综合效果。

我们也做过高性能探索路径，CoreMark/MHz 可以到 5.162186，但该路径资源为 6147 LUT，WNS 为 -0.801 ns，已经不满足初赛冻结提交约束。所以这里把它作为后续优化方向，而不是作为本次提交能力。最终提交优先保证 LUT 小于 5000、50 MHz、可上板和可复现。

## 第 14 页：FPGA 适配与上板证据

这一页用于展示 FPGA 原型系统上板证据。左侧预留的是视频位置，可以放入 FPGA 原型系统上板演示视频或对应截图。这个视频主要展示 Vivado 下载、PYNQ-Z2 连线、CP2102 串口连接，以及终端中 UART 实时输出。

右侧是资源和时序结果。目标板卡为 Xilinx PYNQ-Z2，器件为 xc7z020clg400-1。最终实现资源为 4961 LUT、2367 FF、6 个 BRAM 和 15 个 DSP。时序方面，Setup WNS 为 +0.338 ns，Hold WHS 为 +0.059 ns，CPU 时钟为 50.0 MHz。

底部两行日志是上板证据。Vivado 下载日志显示 PROGRAM_OK: xc7z020_1。串口日志连续输出 YH_rv_cpu CoreMark/MHz=4.137461，DMIPS/MHz=2.908287，并带有不断变化的 tick 和 PC。这说明板上运行不是静态截图，而是持续输出的实时状态流。
