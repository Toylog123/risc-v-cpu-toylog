# 组会 PPT 提示词

> 适用场景：把当前项目进度整理成一份中文技术组会 PPT
> 使用方式：将下面整段提示词直接粘贴到 AI PPT / AI 助手 / 多模态生成工具中

## 提示词正文

请基于以下项目事实，生成一份用于实验室组会汇报的中文 PPT，风格要求是“工程化、清晰、可信、不过度包装”，适合向老师和组内同学汇报项目阶段进展。  

### 一、PPT 基本要求

1. 使用中文，页数控制在 `8-10` 页。
2. 整体风格偏“科研工程汇报”，配色以 `白底 + 深蓝/灰蓝` 为主，不要花哨，不要商业路演风。
3. 每页要有清晰标题、3-5 个核心要点、必要的数据表或对比图。
4. 重点强调“冻结基线”和“活跃工作区”是两层状态，不能混写。
5. 不要把尚未完成 full refresh 的结果写成最终结论。
6. 如果需要图表，优先使用：
   - 柱状图：`CoreMark short` 冻结基线 vs 活跃工作区
   - 表格：验证覆盖矩阵
   - 流程图：当前工作收口逻辑
7. 如果能生成备注，请给每页补 2-4 句演讲备注。

### 二、项目背景

项目名称：`YH_rv_cpu`  
项目定位：面向比赛提交的工程化 RISC-V CPU 基线，不是一次性实验核。  
当前目标：维护一套“可复现、可验证、可交接、可继续优化”的比赛级工程。  
赛题允许 CPU 基于 `RV32I` 或 `RV64I`；当前工程已经具备 `RV32/RV64` 双 XLEN 验证能力，但冻结性能和提交主口径仍以 `RV32I + Zicsr` 为主。  

### 三、必须准确体现的时间线

- `2026-04-08`：完成冻结基线收口，具备正式可引用的 CoreMark、回归、FPGA 证据。
- `2026-04-09`：完成赛题答疑整理，并将汇报与初赛提交材料集中到 `01-项目管理/04-汇报与提交材料/`。
- `2026-04-12`：在活跃工作区保留一轮新的分支重定向优化，short CoreMark 有稳定提升，但 strict 长跑和 FPGA 侧 fresh rerun 还没有补齐。
- `2026-04-17`：当前这次组会汇报基于最新核查整理，未发现比 `2026-04-12` 更晚的 fresh build 证据。

### 五、活跃工作区必须写入的结果

当前活跃 RTL 保留的优化是：  
`taken BEQ/BNE decode-stage early redirect with operand-ready gating`

对应的已验证结果：

- `rv32 baseline = 33/33`
- `rv64 baseline = 21/21`
- `rv32 full-ui = 42/42`
- `rv64 full-ui = 54/54`
- CoreMark short：`10862713 cycles`
- CoreMark short score：`0.925186 CoreMark/MHz`
- 同一 short CoreMark 已重复 rerun，结果完全一致

对应的 profile 结论：

- `ex_branch_redirect_cycles: 1235790 -> 1081457`
- `ex_fetch_redirect_valid_cycles: 1504970 -> 1350637`
- `fetch_queue_empty_cycles` 保持 `1504970` 不变

请明确解释：

- 这说明当前收益主要来自 branch redirect 窗口缩短；
- 这不是 queue/reuse 路径优化带来的收益；
- 这版结果目前还不是新的最终冻结基线。

### 六、必须明确说明的边界和风险

1. `2026-04-12` 的活跃工作区结果还没有补完：
   - strict CoreMark fresh rerun
   - `impl50` fresh rerun
   - FPGA-like probe fresh rerun
2. 所以组会上必须把它表述成：
   - “已验证的活跃工作区优化结果”
   - 而不是“新的最终正式成绩”
3. `fence_i` 的通过依赖扩展 UI 口径 `rv32i_zicsr_zifencei / rv64i_zicsr_zifencei`，这不等于冻结比赛 ISA 已升级。
4. 当前 FPGA 状态是 pre-board closure，不是最终实板 signoff。
5. 外部阻塞仍包括：
   - 实体板卡未到位
   - 最终板级 I/O delay 约束未冻结

### 七、推荐的 PPT 结构

请按下面结构生成：

第 1 页：项目概况与本次汇报结论  
- 项目是什么  
- 当前阶段一句话总结  
- 这次组会最想传达的 3 个结论  

第 2 页：项目当前总体完成度  
- 已完成的工程能力  
- 已完成的验证闭环  
- 已完成的文档与提交材料收口  

第 3 页：冻结基线结果总览  
- 用表格列出 CoreMark、回归、FPGA、资源与时序  
- 强调这是当前正式可引用的口径  

第 4 页：活跃工作区最新推进  
- 介绍 `BEQ/BNE decode-stage early redirect` 这个优化  
- 用简洁流程图或逻辑框图说明优化思路  
- 强调有 guardrail 和 red/green 定向证据  

第 5 页：性能改进对比  
- 用柱状图对比 frozen short CoreMark 和 active short CoreMark  
- 给出绝对值和百分比变化  
- 明确指出“当前只确认 short-run 稳定提升”  

第 6 页：验证与正确性状态  
- 用表格列出 `rv32/rv64 baseline`、`rv32/rv64 full-ui` 的结果  
- 说明 `fence_i` 的边界  
- 说明当前活跃 RTL 还未补完 strict / impl / probe  

第 7 页：当前问题、风险与边界  
- 外部阻塞  
- 口径边界  
- 为什么现在不能直接宣布最终升级  

第 8 页：下一步计划  
- 先补 strict CoreMark  
- 再补 `impl50` 和 FPGA-like probe  
- 如果都保持正向，再刷新冻结基线与全部汇报材料  

如果你觉得内容适合扩成 `9-10` 页，可以额外加入：

- 一页“时间线与里程碑”
- 一页“对老师/评审最值得展示的亮点总结”

### 八、文字表达要求

请使用“真实、克制、工程化”的表达，不要使用以下风格：

- 不要写成“重大突破”“全面领先”“性能大幅飞跃”
- 不要把未完成事项写得像已经完成
- 不要省略日期
- 不要模糊“冻结基线”和“活跃工作区”的区别

更推荐的表述方式：

- “已经形成可提交的冻结基线”
- “在此基础上验证出一版新的活跃优化”
- “short CoreMark 结果显示方向有效”
- “是否升级为新的正式口径，还取决于 strict / FPGA 侧 fresh rerun”

1. PPT 标题页标题和副标题
2. 每一页的页标题
3. 每一页的 3-5 个要点
4. 建议插入的图表类型
