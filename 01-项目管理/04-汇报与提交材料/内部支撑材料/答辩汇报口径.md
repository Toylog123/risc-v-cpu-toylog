# 答辩汇报口径

> 更新日期：2026-04-09

## 项目定位

`YH_rv_cpu` 不是一次性实验核，而是面向比赛提交的工程化 RISC-V CPU 基线。当前主线目标是保持“可复现、可验证、可交接、可继续优化”，而不是为了短期提分破坏基线稳定性。

## 设计亮点

1. 主线口径是 `RV32I + Zicsr` 五级流水，同时保留 `RV64` baseline 共线验证。
2. SoC 已打通 `ROM / RAM / UART / timer / DONE` 最小闭环，基础 smoke 全绿。
3. CoreMark short / strict 双口径都已正式化，既能快速比较，也能给出 strict `>=10s` 结果。
4. `impl50` 已在 Nexys A7-100T `50MHz` 路径上形成稳定 pre-board 基线。

## 当前最好讲清楚的结果

- CoreMark short：`0.912472 CoreMark/MHz`
- CoreMark strict：`0.912465 CoreMark/MHz`，`10.959325s`
- `rv32 baseline = 33/33`
- `rv64 baseline = 21/21`
- `rv32 full-ui = 42/42`
- `rv64 full-ui = 54/54`
- `impl50 = 2556 LUT / 2170 FF / 4 BRAM / 0 DSP`
- `impl50 timing = WNS +5.599ns / WHS +0.025ns`

## 优化口径

- 当前保留优化只有三项：`stall_decode = load_use_hazard`、`IMEM_OUTPUT_REG=0`、`DMEM_OUTPUT_REG=0`
- 已经明确拒绝保留的方向包括 `request-cursor`、`pipe-hit`、`redirect 同拍取指`、`FQ-01 ~ FQ-05` 以及 `FQ-06A`
- `FQ-06A` 虽然诊断全绿，但 short CoreMark 零收益，所以主线 RTL 已回退
- fresh profile 显示 `fetch_queue_empty_cycles` 与 `ex_fetch_redirect_valid_cycles` 完全重合，下一步若继续优化，重点应转向 redirect 成本而不是 request-side 微调

## 需要诚实说明的边界

- 当前 `rv32 full-ui` 的 `fence_i` 通过依赖扩展 UI 口径 `rv32i_zicsr_zifencei`，这不是冻结 ISA 升级
- 当前 FPGA 状态是 pre-board closure，不是实板 signoff
- 缺实体板卡时，UART/LED/boot banner 与最终 I/O delay 约束无法闭环

3. 若有新收益，再按单变量方式进入下一轮优化并同步所有文档。
