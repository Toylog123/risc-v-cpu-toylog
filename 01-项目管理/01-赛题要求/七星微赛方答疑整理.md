## 一、说明

- 本文基于赛方公开腾讯文档提取整理。
- 当前能确认的内容分成“已明确结论”和“仍待确认问题”两类。
- 由于原始表格是在线表格，个别单元格提取后带有前缀字符，或只抓到了问题、没有抓到同一行的完整答复；这类内容一律按“待确认”处理，不擅自补全。

## 二、已明确结论

| 主题 | 当前可确认结论 | 对 `toylog_cpu` 的直接影响 |
|------|------|------|
| 描述语言 | `SystemVerilog` 允许使用 | 当前工程继续以 `Verilog` 为主没有问题，后续如需少量 `SystemVerilog` 特性，赛方口径上可接受 |
| 功能仿真工具 | 可以使用 `verilator` 代替 `modelsim` 做快速功能仿真；提交时要包含设计源文件和 `testbench` | 后续可补一套 `verilator` 快速回归链路，但交付物仍要保留完整测试平台 |
| 存储器实现 | 程序存储器和数据存储器“无限制” | 当前最小 SoC 可以继续采用 `ROM / RAM / BRAM` 路线，不必强行引入外部存储 |
| 评分标准口径 | 评分标准“为参考”，可自行拟定性能达标用例 | 不能只盯一条跑分线，文档里要同时讲清功能闭环、验证完整性和性能依据 |
| 流水线深度 | 赛方答复中出现“可以增加流水” | 可以考虑在五级基线之上做细化优化，但“五级是否仍为最低基线”当前仍需保守处理 |
| 官方硬件/服务器 | 官方“不提供”硬件支持；也未提供服务器 | 队伍必须自备开发板和本地/自有环境，不能把关键路径押在赛方资源上 |
| 建模规范 | 口径是“符合硬件描述语言规范” | 代码风格、层次划分、注释和测试平台要走规范化路线 |
| 初赛提交物 | 当前能明确看到“提交设计说明书” | 设计说明书是必备项；其他提交物仍要继续对照题面和后续通知核对 |
| 特权架构 | “不需要实现特权架构指令” | 初赛阶段可继续聚焦 `RV32I` 主链路，不必为了特权级拖慢闭环 |
| 体系结构自由度 | 允许采用更自由的具体电路技术，答复中提到“具体电路技术可以自行设计” | 高级结构并非被明令禁止，但当前仍应以可交付为先 |
| 测试与 `CoreMark` | 暂无统一测试指令；允许对 `CoreMark` 的 `C` 代码做适当修改以提升性能 | 要自建验证程序集，同时保留对 `CoreMark` 修改点的说明，避免口径不清 |
| 数字孪生平台 | 该赛题没有数字孪生平台，相关内容属于竞业达杯 | 当前项目不需要为七星微题目准备数字孪生展示链路 |
| 设计规范与指标 | 必须实现项和选择实现项要说明清楚；设计规范可自行确定；指标参考赛题文档 | 设计文档里必须主动声明“哪些已实现、哪些选做、哪些未做” |
| 存储资源选择 | 可以完全使用 `BRAM`，是否使用 `DDR` 可按自己需要选择 | 当前板级方案可优先走 `BRAM`，降低实现复杂度 |
| 非核心模块 IP | 部分非核心模块可以使用 `Vivado` 自带 `IP` | `UART`、时钟、调试等非核心模块可合理使用工具 IP，但 CPU 核心仍必须自写 |
| 赛段评分 | 当前答疑中给出的口径是“初赛、赛区决赛、总决赛评分标准相同，可以分步提交文档即可” | 文档资产要从一开始持续积累，不要等到后期再补 |

## 三、仍待确认的问题

下面这些问题在当前公开答疑表里能看到提问，但没有抓到足够清晰的正式答复，暂时不能当成已确认规则使用：

- 是否允许整体或部分模块使用 `Chisel`
- 是否会发放 FPGA 开发板供测试
- `5k LUTs` 资源要求是否为硬门槛
- 对 `cache`、乱序等高级设计是否还有附加要求
- 初赛到赛区决赛、赛区决赛到国赛的淘汰率
- 企业命题解析发布时间
- 初赛是否需要中期报告，以及中期报告的内容要求
- 是否允许超出 `RV32I`，例如加入 `M` 扩展以提升 `CoreMark`
- 评分更偏向性能跑分还是功能完整度
- 是否必须预留 `M` / `C` 扩展接口及其接口标准
- 可扩展性设计（如 `32/64` 切换、可配置流水线深度）在评分中的占比
- `CoreMark` 的目标值是软件仿真达到即可，还是 FPGA 上板也必须达到
- “可以增加流水线”到底是指级内流水、增加总级数，还是仅允许不低于五级

## 四、对当前项目的建议

- 当前项目继续坚持 `五级顺序流水 + 最小 SoC + FPGA 闭环`，不要因为“可以增加流水”就提前扩 scope。
- 既然特权架构当前不需要，`toylog_cpu` 近期优先级仍应放在基础指令、异常最小闭环、`UART`、`timer`、`riscv-tests` 和 `CoreMark`。
- 既然 `BRAM` 可用、外部存储不强制，现阶段不要引入 `DDR` 路线。
- 可以补一条 `verilator` 快速仿真链，但不能替代正式提交所需的 RTL、`testbench` 和文档。
- 所有“赛方未明确答复”的点都不要写进对外承诺或内部硬计划，先按保守口径执行。

## 五、建议后续追问的 5 个核心问题

这 5 个问题最影响技术路线，建议后续优先继续追问赛方：

1. 五级流水到底是固定要求，还是允许改成更多级但不少于五级。
2. `CoreMark` 指标是否要求在 FPGA 实板上达到，而不是只在软件环境中达到。
3. `5k LUTs` 是否为硬门槛，未达到时如何评分。
4. 是否允许加入 `M` 扩展，以及这是否会影响题目要求的原创性和评分口径。
5. `Chisel` 是否允许作为部分模块实现语言，以及提交时对源码形式是否有额外要求。
