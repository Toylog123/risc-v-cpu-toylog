## 四、硬性要求

以下要求必须作为项目基线，后续所有规划和任务都不能偏离：

- CPU 必须基于 `RV32I` 或 `RV64I`。
- 必须采用 `五级流水` 基本架构：`IF / ID / EX / MEM / WB`。
- 必须完成至少 `2 项性能优化`。
- 必须完成 `RTL 建模 -> 仿真验证 -> FPGA 上板` 的完整闭环。
- 必须提交完整源码、测试平台、设计文档、性能报告、调试记录、演示视频。
- FPGA 原型频率不得低于 `50MHz`。
- 代码注释率不得低于 `30%`。
- 评分会关注 `CoreMark`、资源占用和完整跑分结果。

## 五、推荐落实项

这些不是单独写在一条硬约束里，但对比赛结果很关键，项目里应默认纳入：

- 以 `RV32I` 为第一阶段保底，稳定后再评估 `RV32M`。
- 接入 `riscv-tests`、定向测试程序和最小自测程序。
- 板级演示至少具备 `UART` 输出和可复现实验流程。
- 最小 SoC 应包含 `ROM / RAM / UART / timer`。
- 文档、图表、日志和截图必须和代码同步维护。

## 六、红线与边界

- 禁止直接复用开源 `RISC-V CPU` 核心 RTL。
- 可以参考公开资料理解结构，但不能直接拷贝核心实现。
- 初赛前不引入乱序、多发射、复杂 `MMU`、`Linux` 启动等高风险特性。
- 任何新增功能都必须先说明是否会拖慢闭环进度。

## 七、本项目对应口径

### 1. 当前技术基线

- 指令集：`RV32I`
- 结构：`32 位、单发射、五级顺序流水`
- 当前工程状态：`toylog_cpu` 已有五级流水第一版
- 当前主线：补齐 `CSR / timer / trap`、最小 SoC、`riscv-tests`、`CoreMark`、FPGA

### 2. 初赛必须做到

- CPU 主链路稳定运行
- `UART`、`timer` 可用
- `riscv-tests` 基本跑通
- `CoreMark` 数据可提交
- FPGA 上板可复现
- 两项性能优化能够讲清楚做了什么、效果如何

### 3. 初赛不作为硬目标

- 乱序执行
- 多发射
- `Linux`
- 复杂 `MMU`
- 需要大量生态移植才能验证的高风险功能

## 八、后续检查口径

后面每次评审，都用这五个问题做检查：

1. 有没有偏离五级流水主线。
2. 有没有偏离初赛闭环目标。
3. 有没有明确两项性能优化。
4. 有没有可复现的验证和板级路径。
5. 有没有碰原创性红线。
