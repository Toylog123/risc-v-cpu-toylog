# CICC1003618 初赛评委 QA 完整题库

本文从评委视角准备问题和回答，覆盖易、中、难三档。现场回答时优先使用冻结提交口径：`4.137461 CoreMark/MHz`、`2.908287 DMIPS/MHz`、`4961 LUT`、`50.0 MHz`、`WNS +0.338 ns`、`PROGRAM_OK + UART 实时输出`。

## A. 项目定位与赛题理解

### A01 易：你们这个作品解决的是什么问题？

答：我们实现了一颗面向 PYNQ-Z2 FPGA 原型验证的 RISC-V 软核 CPU。它不是只停留在 RTL 仿真，而是完成了从 CPU 架构、RTL、TestBench、性能基准、Vivado 资源时序到物理板下载和 UART 输出的闭环。

### A02 易：为什么说它符合七星微赛题要求？

答：赛题要求 CPU 原型支持 RV32I/RV64I 之一、主频不低于 50 MHz、提交设计文档、验证报告、源码、FPGA 原型系统和演示视频。我们的冻结路径支持 RV32I，PYNQ-Z2 上 50.0 MHz，提交了技术说明书、性能与验证报告、源代码、bitstream、资源/时序报告、上板日志和视频。

### A03 中：你们的设计目标是什么？

答：目标是在 LUT < 5000 的资源约束内做一个可综合、可验证、可上板的五级流水 RISC-V CPU，同时通过热点 ISA 扩展和低成本控制流优化提升 CoreMark 和 Dhrystone 表现。

### A04 中：你们最核心的设计思想是什么？

答：基线指令正确、热点扩展增强、控制路径受限前移。也就是先保证 RV32I 五级流水正确，再保留 Zmmul 和 Bitmanip 热点子集，最后把跳转重定向等低成本控制逻辑提前化，同时保留后级校验。

### A05 难：你们和普通教学五级流水 CPU 的区别在哪里？

答：普通教学核通常强调功能跑通。我们的工作进一步做了三件事：第一，围绕 benchmark 热点保留 Zmmul、Zba/Zbb/Zbs；第二，做 JAL early redirect、forwarding、stall/flush 协同等控制优化；第三，在 PYNQ-Z2 上完成资源、时序、bitstream、UART 输出和性能报告闭环。

## B. CPU 架构与流水线

### B01 易：为什么采用五级流水？

答：五级流水在资源、时序、可解释性和验证复杂度之间比较均衡。对于 PYNQ-Z2 和 LUT < 5000 的边界，它比乱序或复杂超标量结构更适合作为初赛冻结提交路径。

### B02 易：五个阶段分别做什么？

答：IF 取指和维护 PC，ID 译码和寄存器读取，EX 处理 ALU、乘法、位操作和分支校验，MEM 处理访存，WB 完成写回。

### B03 中：数据冒险怎么解决？

答：主要依靠 forwarding 和必要 stall。forwarding 把后级计算结果尽早送回前级，减少等待；对于不能通过前递解决的相关场景，再由 hazard unit 触发停顿。

### B04 中：控制冒险怎么解决？

答：采用 redirect、flush 和后级校验。ID 阶段只提前处理低成本、可确定目标的跳转/分支路径，EX 阶段保留完整校验和兜底重定向，避免提前化破坏正确性。

### B05 难：ID 阶段提前判断会不会拉长关键路径？

答：这是设计时重点控制的风险。我们只在 ID 做低成本判断和目标生成，不把复杂预测器或大规模状态表放进去。最终实现后 WNS 为 +0.338 ns，说明在当前 50 MHz 目标下关键路径可收敛。

### B06 难：为什么不采用乱序执行？

答：乱序执行需要重命名、发射队列、保留站、ROB 或类似提交机制，会显著增加 LUT、状态空间和验证复杂度。对本赛题的 LUT < 5000 和初赛上板要求来说，五级顺序流水加热点优化更稳妥。

### B07 难：JAL early redirect 的收益如何证明？

答：报告中给出 profile 和波形证据。JAL early redirect 后 EX 阶段 JAL redirect 降为 0，对应场景下跳转目标提前进入取指路径，减少前端无效取指。波形页也展示了 Fetch Redirect 与 EX Redirect 的关系。

## C. ISA 扩展与性能热点

### C01 易：提交路径支持哪些 ISA？

答：基础是 RV32I，提交路径加入 Zmmul、Zba/Zbb/Zbs，并包含 JAL early redirect 等微结构优化。Zbc、XThead、IDBR 是探索路径，不进入冻结 bitstream。

### C02 易：有没有 C 扩展？

答：没有把 C 扩展作为初赛提交能力。当前提交材料中不宣称 C 扩展已实现。

### C03 中：为什么做 Zmmul，不做完整 M？

答：Zmmul 保留乘法，收益明确；完整 M 中的除法硬件代价大、时序压力高。对 CoreMark/Dhrystone 和 LUT < 5000 约束来说，保留乘法、关闭除法是更合适的取舍。

### C04 中：Bitmanip 对性能有什么帮助？

答：Zba/Zbb/Zbs 面向地址生成、逻辑融合、位提取和位操作热点。它可以减少部分软件指令序列长度，降低流水线执行压力，对 CoreMark/Dhrystone 这类整数程序有帮助。

### C05 难：Zbc/XThead/IDBR 都做过，为什么不提交？

答：这些路径用于探索性能空间，但最终配置必须满足 LUT、时序、上板和复现。高性能探索路径 CoreMark/MHz 更高，但完整实现超出资源和时序边界，所以不进入冻结提交。

### C06 难：你们是否完整支持 Bitmanip 标准？

答：不是完整 Bitmanip 全标准，而是提交路径保留 Zba/Zbb/Zbs 中对本项目收益明确、资源可接受的子集。现场回答要强调“子集”和“可验证证据”，不要扩大口径。

## D. 性能指标与优化路径

### D01 易：最终 CoreMark/MHz 是多少？

答：冻结提交路径为 4.137461 CoreMark/MHz，对应 2416941 ticks 和 413.746136 iter/s。

### D02 易：最终 DMIPS/MHz 是多少？

答：冻结提交路径为 2.908287 DMIPS/MHz，对应 510986 Dhrystones/s，运行 10 次。

### D03 中：CoreMark 和 Dhrystone 有什么区别？

答：CoreMark 覆盖链表、矩阵、状态机和 CRC 等综合整数路径；Dhrystone 更偏经典整数和库函数路径。两者独立运行、独立报告，不能互相换算。

### D04 中：优化前后提升如何？

答：CoreMark 从早期 RV32I smoke 基线 0.912472 CoreMark/MHz，到参考路径 4.060276，再到冻结提交 4.137461。Dhrystone 从 1.015253 DMIPS/MHz 提升到 2.908287，提升约 186.45%。

### D05 中：主要收益来自哪里？

答：CoreMark 主要来自流水线控制、Zmmul、Bitmanip 子集和 JAL early redirect 的综合收益；DMIPS 提升更明显，主要来自 Dhrystone 构建与库函数路径优化，同时叠加硬件热点优化。

### D06 难：Dhrystone 优化是不是软件作弊？

答：不是隐瞒式优化。构建选项和日志在报告中写明，原始基准源文件不被直接改写，只在生成构建副本中去除 no-inline 限制。我们把它作为独立 Dhrystone 构建口径报告，不和 CoreMark 混用。

### D07 难：CoreMark 是不是严格认证口径？

答：不是 EEMBC 认证口径，而是本项目可复现仿真短运行口径。报告值由 raw ticks 和固定时钟参数解析，CRC 保留在日志中，用于同一环境下的优化对比和提交说明。

### D08 难：为什么不提交 5.162186 CoreMark/MHz？

答：因为该路径资源为 6147 LUT，WNS -0.801 ns，不满足 LUT < 5000 和时序收敛要求。初赛提交强调真实可上板和可复现，所以冻结的是 4961 LUT、WNS +0.338 ns 的路径。

## E. FPGA、资源时序与上板

### E01 易：使用什么板卡？

答：Xilinx PYNQ-Z2，器件为 xc7z020clg400-1。

### E02 易：主频是多少？

答：CPU 时钟 50.0 MHz，满足赛题不低于 50 MHz 的要求。

### E03 易：资源占用是多少？

答：实现后资源为 4961 LUT、2367 FF、6 BRAM、15 DSP。

### E04 易：时序是否收敛？

答：收敛。Setup WNS 为 +0.338 ns，Hold WHS 为 +0.059 ns。

### E05 中：为什么 LUT 接近上限？

答：因为在 RV32I 五级流水基础上保留了 Zmmul、Zba/Zbb/Zbs 和控制流优化，同时要包含 SoC、ROM/RAM/MMIO、UART 等板级逻辑。我们最终选择的是满足上限和时序的配置。

### E06 中：怎么证明已经上板？

答：有 Vivado 下载日志 `PROGRAM_OK: xc7z020_1`，有 PYNQ-Z2 和 CP2102 连接说明，有上板视频，有 UART 连续输出指标流和 tick/PC 变化。

### E07 中：UART 连接怎么做？

答：PL 软核 UART 走 Pmod B，外接 3.3V USB-UART。CP2102 RXD 接 JB1/Y14，GND 共地，串口参数 115200 8N1。Micro-USB/JTAG 用于 Vivado 下载和器件识别。

### E08 难：UART 输出是否等同于板上完整跑完 CoreMark？

答：不完全等同。UART 输出用于展示冻结 bitstream 在板上持续运行和输出状态。完整性能数值来自仿真日志和性能报告；上板证据证明该设计能在真实 FPGA 上下载运行。我们把这两个证据分开说明。

### E09 难：为什么不提高到 100 MHz？

答：赛题要求不低于 50 MHz。当前冻结路径优先满足 LUT < 5000、50 MHz、正时序和上板。进一步提高频率需要更多时序优化和复验，初赛阶段选择稳定可提交版本。

## F. 验证、波形与覆盖

### F01 易：验证体系有哪些层？

答：指令/模块定向测试、SoC smoke、性能基准、FPGA 实现与下载四层。

### F02 易：哪些测试 PASS？

答：SoC smoke、Zmmul、Bitmanip 最终子集、Branch/JAL、CoreMark、Dhrystone、Vivado synthesis/implementation、PYNQ-Z2 PROGRAM_OK。

### F03 中：关键波形展示了什么？

答：展示 ID 分支快速重定向、JAL 快速重定向和 load-use 快速路径。观察 Fetch Redirect、EX Redirect、Stall Decode 等信号，证明控制优化可观察、可解释。

### F04 中：指令覆盖率如何说明？

答：覆盖 RV32I 基础指令、Zmmul、Zba/Zbb/Zbs、非法/非支持路径，以及分支、Load-use、Trap、Timer 等关键场景。当前以定向回归和 benchmark 覆盖为主。

### F05 难：有没有形式化验证？

答：当前初赛提交没有把形式化验证作为主证据，主要采用定向 TestBench、SoC smoke、性能基准、波形检查和 FPGA 实现上板。后续可以引入 riscv-formal 或等价检查增强证明力度。

### F06 难：分支预测准确率是多少？

答：我们的实现不是复杂动态预测器，因此不包装成通用百分比准确率。报告口径是定向诊断 PASS，JAL early redirect 后 EX JAL redirect 计数降为 0，相关错误事件未检出。

## G. 代码、复现与工程规范

### G01 易：源码目录怎么组织？

答：`rtl/` 是 RTL，`tb/` 是 TestBench，`sw/` 是测试程序和 benchmark 端口，`fpga/` 是 PYNQ-Z2 约束与 Vivado 脚本，`scripts/` 是构建、仿真、统计和日志整理脚本。

### G02 易：注释率是多少？

答：保守源码口径 31.87%，RTL/TestBench/Verilog 口径 30.22%，满足不低于 30% 的要求。

### G03 中：怎么复现？

答：按源代码 README 和运行环境说明进入脚本入口，可以运行仿真、性能基准、注释率统计和 FPGA 相关脚本。上板复现使用提交的 bitstream、PYNQ-Z2、CP2102 串口连接和 Vivado 下载流程。

### G04 中：如果评委要看某个模块，应该怎么解释？

答：从分层说：CPU Core 负责流水线和执行控制；SoC 层负责 ROM/RAM/MMIO；FPGA 顶层负责时钟、复位、UART、LED 和引脚约束。新增指令通常要闭合 decoder、执行路径和 TestBench。

### G05 难：你们如何保证参数化探索不会污染提交路径？

答：最终 bitstream 使用冻结配置，探索路径通过参数开关和回归记录保留，不在提交 bitstream 中启用。PPT 和报告中也明确区分冻结路径和探索路径。

## I. 未来工作与提升方向

### I01 易：下一步最想做什么？

答：进一步降低 LUT，给提交路径留更多余量；同时把高性能探索路径中的有效机制重新做面积优化，争取在 5000 LUT 内收敛。

### I02 中：怎么继续提升 CoreMark？

答：从三个方向：更精细的控制流优化、可关闭的热点指令扩展筛选、存储访问路径优化。每一项都要同时检查 LUT、WNS 和上板可复现。

### I03 中：怎么继续提升 DMIPS？

答：继续分析 Dhrystone 热点，包括函数调用、字符串/库函数路径、常量传播、乘法和位操作模式，同时保持构建口径透明，避免只做不可解释的软件改写。

### I04 难：如果复赛资源约束改变，你们会怎么升级？

答：如果 LUT 预算更宽，可以考虑更完整的 Bitmanip 子集、轻量分支预测、I-cache/D-cache 或更深的前端优化；如果仍然严格限制资源，则优先做逻辑复用、控制路径简化和综合策略优化。
