## Root Timing Picture

- `impl136` closes post-route timing, but with small margin: `WNS +0.017 ns`.
- Its worst path is route-dominated and runs from DCache tag state toward ID/EX
  control/data registers, not the original 41-level PC-select path.
- Directly enabling not-taken fold reintroduces a DCache/MEM-to-front-end
  combination path into IF/ID instruction selection.
- RC256 not-taken fold has attractive score and area, but fails synthesis timing
  even with fold EX/MEM load-use forwarding disabled.

## Do Not Repeat

- Do not retry `REDIRECT_CACHE_XOR_INDEX=1` on the current `impl136` family
  unless the RTL changes first; measured result was lower at `4.285487`.
- Do not directly enable `ENABLE_ID_BRANCH_NOT_TAKEN_FOLD=1` in the current
  same-cycle form; `synth132`, `synth133`, and `synth140` all fail timing.
- Do not use old `fast115`/`fast121` high-fold scores as candidate evidence;
  matching implementation failed badly.

## Recommended Next Work

1. Keep `impl136` as the current reportable strict routed candidate.
2. If optimizing performance, work on an RTL-decoupled not-taken-fold variant:
   register the not-taken fold request and/or next-cache lookup so DCache/MEM
   state does not feed IF/ID instruction or PC selection in the same cycle.
3. Before any full implementation, run strict CoreMark fast gate and synthesis.
   Only promote to route if CoreMark/MHz stays above the current target and
   synthesis timing is near closure.
4. After the candidate is stable, generate `impl136` or successor bitstream,
   then collect PROGRAM_OK, UART capture, and board video evidence.
