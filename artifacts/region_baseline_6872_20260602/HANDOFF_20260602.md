## 1. 项目是什么

## 4. 当前阻塞与风险

| 风险 | 说明 | 处理原则 |
|---|---|---|
| post-route timing 未闭合 | WNS 为 `-10.360 ns` | 不得声称该基线已 timing-closed |
| bitstream 不是参赛镜像 | bitstream 虽生成，但时序失败 | 仅作为诊断产物，不用于上板证据 |
| CPU25 demo 尚未上板 | bitstream 已 timing-closed，但缺 PROGRAM_OK/UART/video | 可称为 timing-closed demo bitstream，不能称为 board-proven |
| CoreMark 运行不是 10 秒口径 | 当前为 full workload short-run，CRC 通过 | 公开严谨报告需要补充 >=10 秒 run |
| 未跟踪历史实验文件很多 | `artifacts/fpga_valid_20260518/` 下有大量旧实验日志 | 不要批量 stage，只提交本轮明确相关文件 |
| 无关本地改动 | `YH_rv_cpu/scripts/resolve_python.bat` 为既有 dirty file | 不要纳入冻结提交，除非用户明确要求 |

## 5. 下一步最值得做的 3 到 5 项

| ID | 任务 | 验收标准 |
|---|---|---|
| R01 | 缩短 MEM/DCache 到 PC 的控制路径 | 新 exact-ROM variant WNS 明显改善，CoreMark/MHz 仍高于 4.3 |
| R02 | 参数化实验关停高代价前端路径 | 对比 `load-use spec`、`regular redirect-cache lookup`、`NT-load fold` 的 CoreMark/LUT/WNS |
| R03 | 保持严格 benchmark 口径 | 不修改 `core_list_join.c`、`core_matrix.c`、`core_state.c`、`core_util.c`、`core_main.c` |
| R04 | 使用 CPU25 demo bitstream 做首次上板演示 | PROGRAM_OK，UART 出现 `PERF_DEMO PASS checksum=0xe727358b` |
| R05 | 补充 UART/上板证据 | PYNQ-Z2 串口输出和演示视频与 timing-closed bitstream 对齐 |

建议优先尝试的硬件方向：

1. 关 `ENABLE_DCACHE_LOAD_USE_SPEC`，观察是否切断 DCache probe 到 `stall_decode`/PC 的链。
2. 关 `ENABLE_REDIRECT_CACHE_REGULAR_LOOKUP`，观察是否切断 regular cache hit 到 PC 的组合链。
3. 两者都关，确认最小可 timing 改善组合。
4. 如 CoreMark 掉分过多，再考虑将相关命中信号寄存化或拆分 PC 更新条件，而不是恢复长组合路径。
